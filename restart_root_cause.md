## 4. systemd 정책 적절성

```
[Service]
Type=simple
Restart=on-failure
RestartSec=30
StartLimitInterval=300
StartLimitBurst=3
```

- **Restart=on-failure**: SIGTERM(수동)은 재시작 안 함, 크래시만 재시작 → 크래시 0건이라 시스템 자동 재시작은 실제로 0회 발생
- **StartLimitBurst=3**: 크래시 루프 방지 — 5분 내 3회 연속 실패 시 재시작 중단 (실제 도달 0회)
- **RestartSec=30**: 재시작 대기 30초 — 적절

→ **systemd 정책은 완벽히 적절**. 크래시가 없어서 자동 재시작 로직이 한 번도 작동 안 함.

## 5. 근본 원인 판정

### 🟢 재시작 131회 전부 **수동 배포 재시작** — 운영 이슈 아님

- 크래시 0건, OOM 0건, 네트워크 크래시 0건, StartLimit 도달 0건
- v18.0에서 71회, v18~20.x 버전 전환으로 모두 설명됨
- 최근 v20.9.5 배포 (Apr 24 04:02) 이후 2시간째 안정

### 그러나 개발 루틴 과정에서 발생한 **부작용**은 실존

1. **AR2가 재시작마다 `candles_since_retrain` 리셋** → 30캔들 재학습 영영 못 돌림
   - 재시작 간격이 30캔들(4H × 30 = 5일) 보다 짧으면 카운터 축적 불가
   - Apr 19~24 간격: 40분~60시간 (주로 수시간) → 5일 간격 달성 0회
   - **결과: 최근 5일간 30캔들 주기 재학습 0회 발동** (Regime 전환 트리거 1회만)

2. **`consecutive_train_rejects` 재시작 시 초기화 가능성**
   - 4/24 02:40 AR3 거부 → csv=1 vs status=0 불일치
   - 재시작이 AR3 가드(5회 연속 거부 시 강제 채택) 회피 경로가 됨

### 결론

**재시작 자체는 문제 아님**. 문제는 **재시작 시 카운터/상태 리셋이 정책 의도와 어긋남**.

## 6. v20.9.6 패치 항목 확정

기존 "재시작 잦음" 우려는 단순 개발 루틴이며, v20.9.5 배포 이후 자연 감소 예상. 그러나 **재시작 빈도와 무관하게** 아래 항목은 수정 필요:

### 필수 패치

1. **AR2 리셋 정책 재검토**:
   - 현행: 재시작 시 `candles_since_retrain` 무조건 0 리셋
   - 개선: 재시작 시 **이전 값 유지**. 30캔들 주기 재학습의 본래 의도 복원
   - 영향: 재학습 빈도 증가 (장기 PF 개선 기대)

2. **`consecutive_train_rejects` 영속화**:
   - 현행: status.json 저장/복원 경로 어디선가 누락 (4/24 이후 status=0, csv=1 불일치)
   - 개선: 재시작 시 status.json 복원값 사용 + AR3 가드 회피 방지
   - 영향: AR3 5회 연속 거부 가드 실제 작동 보장

### 선택 패치 (별건 고려)

3. **`check_ai_gate` dead code 제거** (#49 결론)
4. **텔레그램 timeout 35s → 10s** (읽기 응답 대기 단축, 영향 미미)
