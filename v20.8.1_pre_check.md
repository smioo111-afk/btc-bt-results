## 2. AR2 — "Restart 자동학습 OFF" (의미 재정의)

### ⚠️ 사양과 실제 코드의 불일치

**사양**:
> Restart 시 학습 스킵, `needs_retrain`이 첫 호출 시 무조건 False 반환

**실제 코드 분석**:
- `needs_retrain` (L1371-1389)은 두 조건만 있음:
  1. `not self.trained` → True (초기 학습)
  2. `new_candles >= 30` → True (Periodic)
- "Restart" 자체는 트리거가 **아님**

**과거 사례 재검토** (`ai_retrain_analysis.md` Restart 분류 2건):
- **04-10 21:00**: 봇 v18.9→19.0 재시작 후 첫 루프(L3218)에서 `_candles_since_retrain` 영속값이 ≥30 → Periodic 발동
- **04-15 07:12**: 동일 패턴

→ 진짜 원인은 **`_candles_since_retrain` 영속 + 재시작 후 첫 루프 즉시 Periodic 발동**.

### 권장 패치 (Option A — 권장)

`_load_status` 마지막에:
```python
# v20.8.1 AR2: 재시작 시 candles_since_retrain 리셋
# (영속값이 이미 30+이면 첫 루프에서 즉시 Periodic 트리거되는 문제 방지)
if ai_engine.trained:  # 모델 파일 존재
    if default.get("candles_since_retrain", 0) > 0:
        logger.info(f"AR2: candles_since_retrain {default['candles_since_retrain']}→0 리셋 (재시작)")
        default["candles_since_retrain"] = 0
```

또는 더 보수적으로:
```python
# 30 미만이면 그대로, 30+이면 절반으로 — 즉시 트리거 방지하되 곧 학습 가능
saved_count = default.get("candles_since_retrain", 0)
if saved_count >= RETRAIN_MIN_CANDLES:
    default["candles_since_retrain"] = RETRAIN_MIN_CANDLES // 2  # 15
```

### 권장 패치 (Option B — 사양 그대로 해석)

`needs_retrain`에 첫 호출 플래그 추가:
```python
def needs_retrain(self, new_candles: int = 0) -> bool:
    if not self.trained or self.last_train_dt is None:
        self._retrain_reason = "초기 학습"

    # v20.8.1 AR2: 재시작 후 첫 호출은 스킵 (모델 이미 로드됨)

        self._first_check_done = True
        return False
    if new_candles >= RETRAIN_MIN_CANDLES:
        ...
```

### 결정 요청

→ **사용자 결정 필요**: Option A (단순 리셋, 권장) vs Option B (사양 직역).

차이:
- Option A: 다음 30캔들 (5일) 후 정상 Periodic 발동
- Option B: 동일하나 첫 트리거만 한 번 스킵 (영속값 유지)

## 3. AR3 — 모델 롤백 가드

### 코드 위치
- `_do_train`: L1234-1343
- 모델 저장: L1303 `joblib.dump(xgb_new, XGB_PATH)`
- 메타 저장: L1306 `self._save_ai_meta()` → status.json `ai_*` 갱신
- XGB_PATH: `btc_model_xgb.pkl` (이미 백업본 `.bak_phase1` 존재)

### 패치 흐름 (의사코드)
```python
def _do_train(self, df):
    # 1. 학습 전 상태 백업 (메모리)
    old_state = {
        "model": self.xgb_model,
        "trained": self.trained,
        "last_train_dt": self.last_train_dt,
        "precision": self.precision,
        "recall": self.recall,
        "test_accuracy": self.test_accuracy,
        "profit_factor": self.profit_factor,
    }
    has_old_model = self.trained and old_state["profit_factor"] > 0

    # 2. 학습 (기존 로직 그대로)
    ... new_prec, new_pf, new_acc 계산 ...

    # 3. 롤백 가드 (AR3)
    accepted = True
    reject_reason = ""
    if has_old_model and ENABLE_ROLLBACK_GUARD:
        if (new_pf < old_state["profit_factor"] - PF_REJECT_THRESHOLD and
            new_prec < old_state["precision"] - PREC_REJECT_THRESHOLD):
            accepted = False
            reject_reason = (f"PF {old_state['profit_factor']:.2f}→{new_pf:.2f} "
                             f"+ Prec {old_state['precision']:.3f}→{new_prec:.3f}")
            self.status_consec_rejects += 1
            if self.status_consec_rejects >= MAX_CONSECUTIVE_REJECTS:
                # 강제 채택
                accepted = True
                reject_reason += f" → 강제채택({MAX_CONSECUTIVE_REJECTS}회 한도)"
                self.status_consec_rejects = 0

    # 4. 채택/거부 처리
    if accepted:
        self.xgb_model = xgb_new
        self.precision = new_prec
        self.profit_factor = new_pf
        ...
        joblib.dump(xgb_new, XGB_PATH)
        self._save_ai_meta()  # ai_last_train_dt 갱신 → Shadow 동기화 트리거
        self.status_consec_rejects = 0
    else:
        # old_state 유지, 디스크 미변경
        # ai_last_train_dt 미갱신 → Shadow 자동으로 동기화 안 함
        logger.error(f"신모델 거부: {reject_reason}")
        tg_error(...)

    # 5. CSV 영속화 (AR4) — 채택/거부 모두 기록
    self._log_retrain_csv(old_state, new_pf, new_prec, new_acc, accepted, reject_reason)
```

### ⚠️ 거부 기준 (PF AND Prec)의 결정적 한계

사양: `pf_degraded AND prec_degraded` (둘 다 악화 시에만 거부)

**과거 5건 적용 결과** (`ai_retrain_history.csv` 기준):

| 시각 | trigger | ΔPF | ΔPrec | 사양 거부? |
|---|---|--:|--:|---|
| 04-10 22:33 | Periodic | +0.03 | -0.4 | ❌ (PF 양호) |
| 04-13 08:13 | Periodic | **-0.41** | +0.8 | ❌ (Prec 양호) |
| 04-15 07:12 | Restart | **-0.25** | +0.9 | ❌ (Prec 양호) |
| 04-18 05:01 | PerfDegrade | **-0.12** | +0.6 | ❌ (Prec 양호) |
| 04-20 05:45 | Regime | -0.04 | +0.2 | ❌ |

→ **5건 중 0건 거부**. PF 1.20→0.79 (-34%) 같은 명백한 악화도 통과.

이는 본 분석(`bt_candle_count_vs_pf.md`)에서 발견한 **"Validation Prec과 실전 PF가 무관"** 현상 때문. Validation Prec은 거의 모든 학습에서 +0.2~+0.9 개선되므로 AND 조건이 사실상 거부 불가능.

### 거부 기준 대안 (사용자 결정 요청)

| 안 | 기준 | 04-10/13/15/18/20 적용 | 장단점 |
|---|---|---|---|
| **사양 원안** | PF AND Prec 둘 다 악화 | 0/5 거부 | 안전하지만 거의 작동 안 함 |
| **A. PF 단독** | PF < pre - 0.10 | 3/5 거부 (04-13, 04-15, 04-18) | 명확. 거부 빈도 적당 |
| **B. PF OR Prec** | 둘 중 하나만 악화도 거부 | 1/5 거부 (04-10) | Prec noise에 민감 — 너무 많이 거부 |
| **C. PF 가중** | PF < pre - 0.05 OR (PF↓ AND Prec↓) | 4/5 거부 | 보수적 |

→ **권장: 안 A (PF 단독, threshold -0.10)**. 04-20 (PF -0.04)는 통과, 04-13~04-18 같은 큰 악화만 거부.

### consecutive_train_rejects 필드 안전성
- `_load_status` default에 `consecutive_train_rejects: 0` 추가 — 기존 필드와 충돌 없음 (확인됨)
- 5회 연속 거부 시 강제 채택 + tg_error 알림
